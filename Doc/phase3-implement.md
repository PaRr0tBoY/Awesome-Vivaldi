# Ask in Page Phase 3 实现复盘

## 文档目的

本文不是开发计划，也不是通用规范补充，而是对本轮 Phase 2/3 实际实现过程的沉淀：

- 我们最终做成了什么
- 哪些做法在 Vivaldi WebPanel 场景下是有效的
- 哪些问题反复出现，根因是什么
- 后续继续做 Ask in Page 时，哪些流程和判断可以直接复用

---

## 本轮实际完成内容

### 1. 面板架构切换

最初方案走的是 `webview/data URL/iframe` 内部应用模式，后续改为参考 `GlobalMediaControls.js` 的宿主侧注入模式：

- WebPanel 仍通过 `vivaldi.panels.web.elements` 注册
- 真正的 UI 不再依赖 `webview` 内部页面
- Ask in Page UI 直接挂到宿主 panel DOM 上
- 原生 `.webpanel-content` 隐藏，仅保留占位

这一步是后续一切交互稳定的前提。

### 2. 输入区与引用系统

实现了完整的 composer 引用体系：

- 默认当前页面 context card
- `@` 引用网页 / 文件
- `Selected Text` 自动引用卡
- `/` 命令 chip
- 引用卡顶部展示
- 输入区内 inline token 混排
- 发送时按真实顺序组装消息

其中 `/` 命令始终位于最前面，其余引用和正文可混排。

### 3. 消息渲染

消息区已经支持：

- 用户消息气泡
- `/` 命令作为消息开头
- inline 引用 token 作为消息的一部分
- 顶部折叠 context card 组
- AI 回复占位与 `Thinking...`
- `Reading attachments`
- 悬浮复制 / 编辑按钮
- 编辑已有消息并原位覆写该 turn
- 编辑上游消息时，截断其后的整段历史

### 4. 折叠 context card 交互

实现了消息顶部的折叠 / 展开引用区：

- 1 张卡：单卡展示
- 2 张卡：双卡折叠
- 3 张及以上：堆叠展示，点击展开
- 展开后按 panel 宽度自适应网格
- 折叠态支持 hover 动效
- 每张卡都可单独移除

### 5. 编辑态

实现了接近聊天产品的编辑链路：

- hover 消息显示复制 / 编辑按钮
- 点击编辑后，composer 被整条消息覆盖
- 顶部出现 editing banner
- 按回车提交时原位覆盖消息
- 同时清除该消息之后的所有历史 turn
- 在原位置重新生成对应 AI 回复

## 本轮最佳实践

## 1. 在 Vivaldi Panel 里，优先复用“宿主 DOM 注入”模式

只要目标是做一个深度交互型 panel，而不是简单网页展示，优先参考 `GlobalMediaControls.js` 的实现方式，而不是把大量 UI 放进 `webview` 或 `iframe`。

原因：

- `webview` 会带来缓存、焦点、通信、失效 URL、刷新不一致等问题
- 宿主 DOM 能直接访问 Vivaldi 面板结构和 Chrome API
- 排查 CSS/布局问题时，可直接在同一 DOM 树中定位根因

本项目中，很多“行为看起来没更新”的问题，本质上都是旧 iframe/webview 内容没有真正刷新。

## 2. 在 Vivaldi 内做 UI，先怀疑宿主样式冲突，再怀疑自己样式

这次最典型的例子是 `common.css` 中的：

```css
.panel section {
    display: flex;
    flex-direction: column;
    flex: 1 1 auto;
    overflow-x: hidden;
    overflow-y: auto;
    position: relative;
}
```

它直接导致：

- 输入区被拉伸到中间
- dropdown 分组布局异常
- 看似“z-index”问题，实际是父层 flex/overflow 规则被改写

可复用结论：

- 在 `.panel` 内部尽量避免滥用裸 `section`
- 布局异常先查 computed style 和宿主命中规则
- 不要过早把问题归结为 `z-index`

## 3. 样式更新必须支持“覆盖旧 style”，不能只 append 一次

Vivaldi 这类宿主环境里，经常会出现：

- JS 逻辑更新了
- 版本号更新了
- 但旧 `<style>` 仍在生效

这次已经验证：如果 `addStyle()` 只在第一次插入样式，后面所有 CSS 修复都有可能“看起来没生效”。  
因此宿主注入脚本必须允许同 id `<style>` 被重写。

## 4. 能统一成一个组件模型的，不要做两套

默认当前页卡和 `@` 添加卡在前期曾是两种不同 DOM 和样式路径，结果带来一串问题：

- 高度不同
- 对齐不同
- overflow 行为不同
- `@menu` 去重困难
- 删除和同步逻辑复杂

后面统一成同一个 `@card` 模型后，问题明显减少。  
结论：只要语义一致，就优先统一 DOM 结构、状态字段和渲染入口。

## 5. 对“镜像 UI”和“真实输入内容”要做严格分层

顶部 context card 是可视镜像，不是输入流本体。  
输入区 inline token 才是真正决定发送内容、删除行为、复制粘贴行为的“主数据结构”。

一旦把镜像层也纳入键盘删除 / 光标邻接逻辑，就会出现非常诡异的行为：

- 光标在左边，删到了右边
- 光标看似挨着输入 token，实际选中了顶部 mirror card

可复用原则：

- 键盘行为只基于 composer 内真实 token
- 顶部 card 只做状态镜像和鼠标操作
- 二者同步，但不要共享“可选中目标”

## 6. 与其放开整层 `overflow: visible`，不如预留安全区

为了解决 context card 右上角 `x` 被裁切的问题，如果简单把整条容器链都改成 `overflow: visible`，会带来新的问题：

- 多个 card 右侧溢出对话框
- hover 元素破坏整体边界

后面验证更稳的方案是：

- 外层继续裁切
- 在容器内部为“出框按钮”预留安全 padding
- 按钮通过定位和 `transform` 轻微出框

这是在紧凑 UI 中更可控的处理方式。

## 7. 每一版都显示可见版本号，能极大降低“缓存误判”

当 panel 有缓存、旧样式残留、旧 DOM 没销毁时，纯靠“我已经改了代码”很难确认用户看到的是不是新版本。  
右上角版本号 `v1 -> v2 -> ...` 在这次迭代里非常有用，建议保留为长期机制。

## 本轮遇到的关键问题与解决方式

## 1. 对话框始终浮在中间

### 现象

输入区应该贴底，但总是浮在面板中间。

### 根因

- Vivaldi `common.css` 命中了 `.panel section`
- 输入区和 dropdown 分组使用了 `section`
- 被强行套上 `flex: 1 1 auto`

### 解决

- 相关结构改为 `div`
- 明确给输入区 `flex: 0 0 auto`
- 隐藏原生 `.webpanel-content`

## 2. `@` 已触发但 dropdown 不可见

### 现象

- 输入 `@` 后有阴影
- DOM 中 `visible` 已经存在
- 但用户看不到内容

### 根因

不是数据没取到，而是宿主布局 / overflow / section 样式破坏了 dropdown 渲染。

### 解决

- 避开 `.panel section` 样式
- 调整 dropdown 所在层级和 z-index
- 保证分组容器不再被宿主 flex 规则接管

## 3. `/` 命令无限叠加

### 现象

- 可添加多个 `/` 命令
- 每次发送只带走一个
- 剩余 chip 卡在输入框中

### 根因

`/` 命令逻辑最初没有做“单实例约束”。

### 解决

- 选择新命令时先清理旧命令
- 发送后统一清空命令状态
- 后续再把 `/` 命令接入和普通 token 一致的键盘删除模型

## 4. 关闭按钮一会儿被裁，一会儿把整排挤出边界

### 现象

- 按钮显示不全
- 改成 `overflow: visible` 后，又导致卡片溢出对话框

### 根因

父层 overflow 链过深，且不同层承担了“裁切”和“视觉溢出”两种矛盾职责。

### 解决

- 恢复外层裁切
- 在需要的位置预留安全区
- 只让按钮视觉上“出框”，而不是让整层都失去边界

## 5. 顶部当前页卡和 `@` 引用卡不一致

### 现象

- 宽高不同
- 对齐不同
- `@menu` 里会出现当前页重复项

### 根因

两者不是同一个组件模型。

### 解决

- 收敛为统一 `@card`
- 当前页也作为同类引用处理
- `@menu` 中对已选项做去重

## 6. 输入时引用卡闪烁

### 现象

每打一字，顶部引用区闪一下。

### 根因

`input` 触发后无差别重建顶部引用区。

### 解决

- 对当前页、selected text、refs 做状态比较
- 无变化时不重绘

## 7. Backspace 误删错误 token

### 现象

- 光标在左边，删掉右边 token
- 光标在最左侧时直接删整条内容

### 根因

- 存在“找不到相邻 token 就默认选最后一个”的 fallback
- 浏览器默认 `contenteditable` 删除行为在边界处接管

### 解决

- 改成严格的“相邻 token”判断
- 删除 fallback
- 在最左侧拦截 `Backspace` 默认行为
- 顶部 mirror card 不参与键盘邻接判断

## 8. 编辑消息后，后续历史没有被截断

### 现象

编辑 B 后，C/D/E 还保留，和真实聊天产品不一致。

### 根因

只更新了当前 turn，没有实现历史覆写。

### 解决

- 编辑已有 turn 时，删除该 turn 之后全部 `.ask-turn`
- 同时清除挂起的 AI 回复计时器

## 可复用开发流程

后续继续开发 Ask in Page，建议直接复用以下流程。

## 1. 新功能先判断属于哪一层

每次加功能前，先归类：

- 宿主 panel 层：挂载、样式、布局、面板生命周期
- 引用镜像层：顶部 cards、折叠展示
- 输入流层：composer token、光标、复制粘贴、键盘删除
- 消息层：用户消息、AI 回复、turn 生命周期
- 数据层：tabs/files/selection/downloads

先分层，再写代码，能明显减少“为了修一个交互，误伤别的层”的情况。

## 2. 遇到 UI 错位，先查宿主 CSS 命中链

推荐检查顺序：

1. 看 DOM 是否在预期层级
2. 看 computed style
3. 查是否命中 `common.css`
4. 再看是不是我们自己的 flex/overflow/z-index

这套顺序比直接猜 `position` 或 `z-index` 更有效。

## 3. 遇到“状态不同步”，先找是否存在双表示

这次很多 bug 都来自“同一份数据有两个可操作表示”：

- 顶部 card
- 输入 token
- 消息 inline 引用

处理方式应该是：

- 只保留一个主状态源
- 其余 UI 全做镜像渲染
- 删除和键盘逻辑只绑定主状态源

## 4. 每个复杂交互都要先定义“状态转移”

例如编辑消息时，真正要定义的不是“点击按钮后做什么”，而是：

- 进入编辑态时，哪些状态会被覆盖
- 提交时，哪些 turn 需要截断
- 取消时，哪些状态要回滚

写清楚状态转移，再落代码，会少很多补丁式修复。

## 5. 每次较大改动后都做可见验证点

这次实践里最有效的验证点包括：

- 右上角版本号
- `@menu` 是否出现
- 当前页卡是否同步
- `Backspace` 是否只删左邻 token
- 编辑上游消息是否截断后续 turn

后续可以继续沿用“每个功能只留 2-3 个高信号验证点”的方式推进。

## 仍可改进的部分

以下不是本轮失败项，而是下一轮可以继续收口的方向。

## 1. token 混排模型还能继续增强

当前已经支持 token 混排、复制、粘贴、删除，但仍建议后续继续加强：

- 更完整的剪贴板序列化协议
- 更稳定的 DOM 到状态反向解析
- 更细的光标恢复策略

## 2. “发送给 AI 的真实 payload”建议单独抽象

目前发送逻辑已经按顺序组装，但后续如果要接真实 AI API，建议单独抽成：

- `display model`：界面展示
- `transport model`：发给 AI 的结构化 payload

这样能避免 UI 迭代时误伤消息协议。

## 3. turn 生命周期可继续标准化

现在已经有：

- 用户消息
- AI 回复
- 编辑覆盖
- 历史截断

后面可以继续统一成明确状态机，例如：

- `draft`
- `submitted`
- `streaming`
- `completed`
- `edited`
- `superseded`

这会让后续接真流式输出更顺。

---

## 推荐保留的工程约定

这些约定是这次实践中证明有效的，建议保留。

- Ask in Page 继续沿用宿主 DOM 注入模式，不回退到 iframe/webview UI
- 每次可见改动递增右上角版本号
- 新增引用类型时，优先接入统一 `@card` / token / message 三层同步链路
- 键盘删除、方向键导航，只作用于 composer 内真实 token
- 任何“已选引用”都要从 `@menu` 中去重
- 任何编辑旧消息的操作，都必须考虑是否截断后续历史

---

## 一句话结论

这次 Ask in Page 的 Phase 3 真正可复用的经验，不是“又修了多少 UI 细节”，而是已经逐步形成了一套适用于 Vivaldi Panel 的实现方法：宿主注入、统一引用模型、镜像与输入分层、状态驱动的消息历史管理。后续继续做新功能时，只要不偏离这条线，复杂度会明显低很多。
